# -ExhiTrack-
我要为自己开发一个个人观展档案管理系统，解决我看完展览后大量照片堆积、缺乏动力整理的痛点。 核心价值主张：把"看完展回家"到"得到一份完整观展档案"的整理流程，从数小时压缩到 10 分钟内。
> 本文档用作与 AI 编码助手（Claude / Cursor / Copilot 等）协作开发时的**主提示词**。
> 使用方式：开新会话时把本文档作为初始上下文喂给 AI；后续每次提需求时引用对应章节即可。
---

## 一、项目背景与目标

### 1.1 产品愿景

我要为自己开发一个**移动端个人观展档案管理 App**——它不仅是一个"看展记录工具"，更是一个**由视觉 AI 驱动的个人文博知识引擎**。

我的最终愿景是：**让每一张展厅里随手拍下的文物照片，都能被 AI 自动"看懂"、被联网检索到的权威资料"喂饱"、被我的个人感受"打上烙印"，最终沉淀为一座属于我个人、可全库检索、可终身积累的线上文博博物馆。**

### 1.2 痛点与产品定位

我是一个频繁观展的爱好者，常去博物馆和美术馆，习惯给文物拍大量照片、留下当下的感受和思考。但**回家后面对几百张零散照片，缺乏时间和动力按展览叙事重新整理**：

- 📷 照片散落在相册里，分不清哪些是哪场展、哪个文物
- 🏷️ 展签信息拍得模糊或被玻璃反光，回家也懒得逐张辨认
- 🧠 当时的感悟想法零碎，过几天就忘
- 🔍 看过的同类文物在不同博物馆出现过，无法横向联系
- 📚 想深入了解某件文物，要在多个网站之间反复搜索切换

久而久之，看过的展览只剩下手机里没人翻的照片，毫无沉淀。

**这个产品要做的，是把"看展 → 拍照 → AI 识别 → 联网增强 → 个人化整理 → 长期沉淀"的完整闭环极致简化**，让每一次观展都能自动生成一份图文并茂、贴合策展叙事、有个人感受、附带权威背景资料、可检索可分享的**专属观展档案**。

**三大核心价值主张**：

| 价值 | 表现形式 |
|------|---------|
| **现场零负担** | 看展时只管拍照和录语音，不打断观展节奏；不需要现场动脑分类 |
| **AI 自动加持** | 视觉 AI "看懂"每张照片 + 联网检索权威资料 + LLM 生成结构化档案 |
| **长期可沉淀** | 所有观展记录沉淀为可全库检索、可时间线回顾的个人文博知识库 |

**用户范围**：仅自用，不对外发布，不考虑多租户、不考虑商业化、不考虑大规模并发。但产品本身要足够好，至少自己用着舒服。

### 1.3 主要使用场景

- **现场场景**（高优先级）：在展厅看完一件文物后立刻拍照、录语音、加感受；展览看完后在咖啡区或路上立刻开始整理
- **路上场景**（高优先级）：地铁、公交、网约车上单手操作整理，等待 AI 处理结果时翻翻已有档案
- **回家场景**（中优先级）：偶尔在家用手机做更精细的归档和导出（不优化大屏体验，不做平板适配）
- **回顾场景**（长期价值）：随时打开 App 查询"我去年看过的所有唐代陶俑"、回顾"故宫看过的所有展览"

### 1.4 完整功能矩阵

> 以下功能分为 **MVP 核心**（必做，10-14 周内完成）和 **拓展功能**（v2+ 视使用情况增量开发）。
> 视觉 AI 识别 + 联网检索（B 模块）是产品最核心的差异化能力，必须 MVP 阶段就跑通。

#### A. 智能拍摄模块（现场用，从源头降低整理成本）

**MVP 核心**：
- **A1. 文物拍摄优化**：自定义相机界面，针对展厅环境优化（防抖、暗光对焦、最大化曝光锁定）；快门按钮巨大易触
- **A2. 一键多拍归组**：拍摄时自动连拍多张归为一组（文物本体 + 展签 + 细节），后台自动判断哪张是展签、哪张是本体
- **A3. 实时展签 OCR**：拍摄完成后当场调用 OCR 识别展签文字并结构化（朝代 / 品类 / 出土地等），无需回家二次处理
- **A4. 展览自动归类**：拍摄前选择或新建当前展览；所有拍摄内容自动归入该展览档案，按拍摄时间排序，自动还原观展路线
- **A5. 实时语音备注**：拍摄后一键录音感受，自动与对应文物绑定，离线缓存、上线后自动转文字
- **A6. GPS 自动识别博物馆**：拍照时记录 GPS，匹配到附近博物馆，自动填充展览的"博物馆"字段

**拓展功能**：
- A7. **玻璃反光提示**：用本地图像处理检测反光严重区域，提示用户调整角度
- A8. **拍摄路径回放**：以时间轴形式可视化整个观展过程的拍摄序列
- A9. **现场快速笔记**：除了语音，支持手写涂鸦或文字便签贴到照片上
- A10. **拍摄构图建议**：基于历史拍摄数据，提示"你拍青铜器倾向正面，建议补一张俯拍"

#### B. 视觉 AI 识别 + 联网检索模块（⭐ 产品核心智能层，MVP 必做）

**这是整个产品的灵魂模块**——它把每张照片从"像素"变成"知识"。

**MVP 核心**：

- **B1. 视觉 AI 文物识别**：
  - 调用多模态视觉 LLM（首选 **通义千问 VL-Max**，备选 GPT-4o vision / Claude Sonnet 4 vision）
  - 直接"看"文物照片，无需依赖展签即可识别：
    - 文物类型（青铜鼎 / 唐三彩 / 宋瓷 / 玉璧 / ...）
    - 形制特征（三足两耳 / 兽面纹 / 莲花口 / ...）
    - 推测朝代和工艺（带置信度）
    - 候选文物名（如"疑似司母戊鼎，置信度 70%"）
    - 视觉风格描述（用于后续语义检索）

- **B2. 视觉 + OCR 双通道融合**：
  把视觉识别结果和展签 OCR 结果交叉验证、互相补充：
  - 展签清晰 → OCR 为主事实，视觉补充工艺细节
  - 展签模糊 / 被玻璃反光遮挡 → 视觉 AI 主导判断
  - 两者矛盾 → UI 上明确标记"⚠️ OCR 认为是宋代，视觉 AI 认为是唐代"，让用户裁决

- **B3. 联网检索权威资料**：
  基于视觉 + OCR 识别结果，自动联网检索权威信息源：
  - **大馆官网藏品库**（MVP 优先适配 Top 10 大馆）：
    - 故宫博物院 (dpm.org.cn)
    - 中国国家博物馆 (chnmuseum.cn)
    - 上海博物馆 (shanghaimuseum.net)
    - 南京博物院 (njmuseum.com)
    - 苏州博物馆、湖南省博、河南博物院、陕西历史博物馆、湖北省博、浙江省博等
  - **百科类**：维基百科中文版（首选，结构化好）、百度百科、中国大百科全书
  - **检索方式**：
    - 优先用文物专名直接检索官网藏品页
    - 失败则用"朝代+品类+特征"作为关键词搜索
    - 用 LLM 判断检索结果的相关性，过滤无关页面
  - **结果处理**：检索到的页面由 LLM 提炼成"扩展知识卡片"（不超过 300 字 + 关键事实点 + 原始链接）

- **B4. 信息可信度分级与溯源**：
  每个字段都标注信息来源，UI 上用不同颜色或图标区分：
  - 🏛️ 博物馆官方（最高可信度）
  - 📖 百科条目
  - 👁️ 视觉 AI 识别
  - 🏷️ 展签 OCR
  - ✏️ 用户手动编辑
  - ⚠️ AI 推测（低置信度）
  
  **绝不让 AI 幻觉冒充事实**——所有字段都必须能点击查看来源链接或原始 OCR 文本。

- **B5. 按需触发 + 成本控制**：
  - 默认只对用户标记为"重点文物"的照片触发视觉 AI（避免动辄几十张照片调用费用爆炸）
  - 提供"批量深度识别"按钮，触发前明确告知预计费用
  - 视觉 AI 结果本地缓存，重复查看不重复调用

**拓展功能**：
- B6. **同类文物推荐**：识别后自动关联"我之前看过的同朝代 / 同品类文物"，建立内部交叉引用
- B7. **跨博物馆藏品比较**：自动检索其他博物馆类似藏品（如"全国各大博物馆的青铜鼎对比"），生成对比卡片
- B8. **学术深读模式**：对感兴趣的文物，一键拉取 CNKI / Google Scholar 上的相关论文摘要
- B9. **文物知识图谱**：随着数据库积累，自动构建朝代 / 品类 / 出土地 / 工艺的关联图谱，可视化呈现
- B10. **以图搜图**：上传一张文物图，找出我数据库里所有相似的文物
- B11. **AI 讲解员**：选中一件文物，AI 用语音生成 2-3 分钟的"博物馆讲解"音频
- B12. **真伪 / 年代辅助判断**：对未知出处的文物（如自家收藏、市集所见），让视觉 AI 给出风格分析（明确不做鉴定，只做学习参考）

#### C. 一键智能整理模块（回家后 10 分钟完成全流程）

**MVP 核心**：
- **C1. 策展叙事还原**：基于展览名 + 拍摄的策展前言照片 + 文物列表，由 LLM 推导策展单元划分（标题 + 描述）；优先尝试联网获取展览官方信息，找不到则纯 LLM 推理
- **C2. 文物 → 单元自动分配**：把每件文物归入最匹配的策展单元，结合 B 模块的视觉识别结果提升准确率
- **C3. 批量去重 / 模糊废片剔除**：客户端用感知哈希 + 模糊度检测自动标记
- **C4. 信息自动补全**：对展签字段缺失的文物，用 B3 联网检索结果自动补全
- **C5. 感受自动润色**：把碎片化语音 / 文字感受 LLM 润色成连贯文本，保留个人语气
- **C6. 智能标签生成**：基于视觉 + 文字内容自动打标签（如"唐代""彩陶""出土于河南"），便于后续检索

**拓展功能**：
- C7. **自动错误修正**：检测明显的分类错误（如"宋代青铜器"会被标记为可疑，因青铜器主流在商周）
- C8. **双向编辑**：AI 整理结果支持 diff 视图，逐条接受或回退
- C9. **重新策展**：手动指定一个主题（如"按工艺类别"），让 AI 重新组织已有文物
- C10. **多展览串联叙事**：自动生成"我看过的所有青铜器主题展"这种跨展览叙事

#### D. 观展档案生成与管理模块

**MVP 核心**：
- **D1. 多格式导出**：
  - 社交平台精简版（小红书风格 9 宫格 + 短文）
  - 个人归档完整版（含所有照片 + 感受 + 联网检索的扩展知识 + 来源链接）
  - PDF 打印版
  - Markdown / Notion / 飞书导入版
  - 全库 JSON 备份
- **D2. 个人文博数据库**：自动建库，按展览时间 / 文物朝代 / 品类 / 博物馆 / 出土地多维分类；支持全文检索（如搜"唐代 青铜器"调出所有相关）
- **D3. 观展统计**：展览数量、博物馆打卡数、文物朝代分布、品类偏好；月度 / 年度自动生成观展报告

**拓展功能**：
- D4. **时间线视图**：按观展时间生成可视化时间轴
- D5. **地图视图**：在中国地图 / 世界地图上标记去过的博物馆，点击展开当地观展记录
- D6. **年度个人观展报告**：年底自动生成"年度观展回顾"长图（类似网易云年度报告）
- D7. **知识卡片闪卡**：把识别出的文物变成记忆卡片，用 SRS 算法帮我记忆经典文物
- D8. **观展兴趣演化分析**：分析我的观展偏好如何随时间变化（如"今年明显更多关注书画"）
- D9. **下一站推荐**：基于历史偏好和位置，推荐可能感兴趣的展览或博物馆

#### E. 共享与协作模块（拓展功能，MVP 不做）

- E1. 与同行好友共享展览档案，合并各自拍摄的照片与备注
- E2. 公开档案为只读链接，分享给社交圈
- E3. 关注其他用户、订阅他们的观展记录
- E4. 协作编辑某场展览的"集体记忆"档案

### 1.5 视觉 AI + 联网检索的工作流（核心架构详解）

这是产品最具差异化价值、也是技术上最复杂的部分，单独说清楚四阶段架构：

**📥 输入**：用户拍的一组照片（通常 2-5 张：文物正面、侧面、细节、展签）

**🔧 第一阶段：本地预处理（手机端，离线完成）**
1. 客户端按感知哈希（pHash）分组——同一文物的多张归一组
2. 自动判断每张照片是"文物本体"还是"展签"（比例 + 文字密度 + 视觉特征判断）
3. 生成缩略图（用于 UI 展示和 AI 调用前的压缩）
4. 提取 EXIF 时间戳和 GPS

**⚡ 第二阶段：双通道并行识别（云端 Edge Function）**

并行调用以节省时间：

- **展签通道**：阿里云 OCR → LLM 结构化抽取（输出 JSON）
- **视觉通道**：把"文物本体"照片喂给视觉 LLM，要求输出：
  ```json
  {
    "type": "青铜鼎",
    "shape": "三足两耳，方鼎形制",
    "decoration": "饕餮纹，云雷纹底",
    "estimated_dynasty": "商代晚期",
    "confidence": 0.85,
    "candidate_names": ["疑似司母戊鼎类型", "或为方鼎一族"],
    "craft_features": ["失蜡法铸造", "高浮雕"],
    "visual_description": "通体青绿色，包浆厚重，纹饰繁复..."
  }
  ```

**🌐 第三阶段：联网检索增强（云端 Edge Function）**

用第二阶段的输出作为查询词，按优先级检索：

1. **首选：博物馆官网藏品库**
   - 用文物专名直接搜索 Top 10 大馆官网
   - 通过 web fetch 抓取藏品详情页
   - 优势：信息权威、有高清图、有专业描述
2. **次选：维基百科中文版**
   - 通过 Wikipedia API 拉取结构化数据
   - 优势：信息标准化、有 infobox
3. **兜底：百度百科 / 中国大百科**
   - 通用知识兜底
4. **可选增强**：CNKI 论文摘要（v2 拓展）

检索结果由 LLM 二次提炼：
- 去除冗余和广告内容
- 提炼关键事实点（生成时间、规格、出土地、馆藏地等）
- 生成 200-300 字的"扩展知识卡片"
- **保留所有原始 URL 用于溯源**

**🔀 第四阶段：信息融合与可信度标注**

把展签 OCR / 视觉识别 / 联网检索三方结果合并到统一的文物记录中：

| 字段 | 来源优先级 | UI 标注 |
|------|----------|--------|
| 文物名称 | 博物馆官网 > 展签 OCR > 视觉 AI 候选 | 显示来源图标 |
| 朝代 | 博物馆官网 > 展签 OCR > 视觉 AI（如冲突标记 ⚠️） | 冲突时双显 |
| 品类 | 视觉 AI > OCR > 百科 | 高置信度即可 |
| 出土地 | 博物馆官网 > 展签 OCR > 百科 | 严格只取权威源 |
| 工艺特征 | 视觉 AI + 百科 + 论文（多源合并） | 列出所有来源 |
| 历史背景 | 博物馆官网 > 百科 > LLM 生成 | LLM 生成需明确标注 |
| 个人感受 | 用户语音 / 文字（LLM 仅润色） | 标注"已润色" |

**🎯 关键设计原则**：

1. **绝不让 AI 幻觉冒充事实**：所有字段必须可溯源到原始来源链接或 OCR 文本
2. **离线兜底**：联网检索失败不影响展签 OCR 和视觉识别本身
3. **成本可控**：
   - 视觉 LLM 调用按需触发，默认不自动跑全部照片
   - 联网检索结果本地缓存（同一文物名 7 天内不重复检索）
   - 用户可在设置中查看"本月 API 调用费用"
4. **用户最终裁决**：所有 AI 结果可一键编辑或回退到原始 OCR / 原始照片
5. **渐进式增强**：基础识别完成后用户即可使用，联网检索结果异步追加（不阻塞主流程）

### 1.6 产品演进路线图

为避免范围蔓延，分阶段演进：

| 阶段 | 时间 | 重点 | 验收标志 |
|------|------|------|---------|
| **MVP** | 10-14 周 | A 拍摄 + B 视觉 AI + 基础整理 + 简单导出 | 能完整跑完一场展览的"拍-识-理-存"闭环 |
| **v1.1** | MVP 后 1-2 月 | B 模块联网检索的覆盖率提升、UI 打磨 | 联网检索命中率 > 60% |
| **v2** | 半年内 | D 模块拓展（地图视图、统计深化）、C 模块跨展览叙事 | 数据库积累 50+ 场展览后的整体体验优化 |
| **v3+** | 一年后 | B 模块图谱、AI 讲解员、E 共享模块 | 视使用情况和热情决定 |

### 1.7 开发模式与周期

**开发模式**：Vibe coding —— 我描述需求，AI 写代码，我在手机上验证后迭代。我具备基础的产品设计能力，但代码能力较弱，因此架构选型、模块边界、技术栈都需要对 AI 生成友好。

**预期周期**：MVP 阶段 10–14 周（视觉 AI + 联网检索是最大的不确定性来源，需预留充分调试时间），按周末 + 工作日晚上的投入节奏。拓展功能根据自用反馈增量开发，不设硬性时间表。

**心态准则**：
- 视觉 AI 识别准确率不会一开始就完美，**目标是 70% 自动 + 30% 手动补全**，不追求 100% 自动化
- 联网检索覆盖率取决于馆方数据开放程度，**做不到全覆盖是正常的**
- 自用产品**不需要打磨极致 UI**，能用、好用即可

## 三、架构原则与约束

1. **离线优先**：所有写操作先落本地 SQLite，再异步同步到 Supabase。展厅信号差、地铁断网时仍可用。
2. **单手友好**：主要操作放在屏幕下半部分，按钮足够大（最小 44pt），关键流程不超过 3 步。
3. **模块边界清晰**：每个功能模块的输入输出都是明确的数据结构，不允许跨模块紧耦合。
4. **失败优雅降级**：任何 AI 服务调用失败都不能阻塞主流程，必须允许用户手动补全。
5. **批处理优先**：所有 AI 调用尽量批处理，避免单条调用消耗流量和电量。
6. **客户端能做的不上后端**：图像压缩、缩略图生成、模糊检测、相似度比对优先在手机本地完成。
7. **数据可导出**：所有数据必须可导出为 JSON / Markdown，避免厂商锁定。
8. **省电省流量**：照片上传默认走 WiFi，蜂窝网络下需用户确认；后台任务遵循系统电量策略。
9. **自用场景，不要过度设计**：
   - RLS（行级安全）默认关闭
   - 不需要复杂权限系统
   - 不需要审计日志
   - 不需要国际化（只做中文）
   - 不需要暗色模式（v2 再做）

## 四、数据模型

### 双层数据架构

**本地层（expo-sqlite）**：完整数据 + 同步状态字段
**云端层（Supabase Postgres）**：完整数据备份 + 跨设备同步源

每条记录有 `sync_status` 字段：`pending` / `synced` / `failed`，由后台同步任务管理。

### 核心表结构（本地与云端一致）

```sql
-- 展览表
exhibitions
  - id (uuid, PK)
  - name (text, 展览名称)
  - museum (text, 博物馆名称)
  - location (text, 城市)
  - start_date (date, 展览开始日期)
  - end_date (date, 展览结束日期)
  - visit_date (date, 我的观展日期)
  - description (text, 展览介绍 / 策展前言)
  - curatorial_units (jsonb, 策展单元 [{title, description, order}])
  - cover_image_url (text)
  - gps_lat, gps_lng (real, 拍摄时的 GPS，用于自动识别博物馆)
  - sync_status (text)
  - created_at, updated_at (timestamptz)

-- 文物表
artifacts
  - id (uuid, PK)
  - exhibition_id (uuid, FK -> exhibitions)
  - curatorial_unit_index (int, 所属策展单元序号，可空)
  - name (text)
  - dynasty (text, 朝代)
  - category (text, 品类)
  - origin (text, 出土地 / 来源)
  - era (text, 具体年代)
  - description (text, 展签原文)
  - structured_info (jsonb, LLM 抽取的结构化字段)
  - photo_local_paths (text[], 本地文件路径)
  - photo_cloud_urls (text[], 云端 URL)
  - thumbnail_local_path (text)
  - thumbnail_cloud_url (text)
  - photo_taken_at (timestamptz)
  - is_blurry (bool)
  - is_duplicate_of (uuid)
  - tags (text[])
  -- 信息溯源（核心新增，每个字段记录来源）
  - field_sources (jsonb)
    -- 形如 {"name": "ocr", "dynasty": "vision", "origin": "wikipedia",
    --       "era": "user_edited", "category": "museum_official"}
    -- 可选值：ocr / vision / wikipedia / baidu_baike / museum_official 
    --        / llm_inferred / user_edited / null
  -- 视觉识别结果（核心新增）
  - vision_analysis (jsonb)
    -- 形如 {"shape": "三足两耳鼎", "patterns": ["饕餮纹", "云雷纹"],
    --       "estimated_dynasty": "商代晚期", "confidence": 0.85,
    --       "candidates": ["司母戊鼎", "大克鼎"], "raw_response": "..."}
  -- 联网检索结果（核心新增）
  - knowledge_card (jsonb)
    -- 形如 {"summary": "...", "key_facts": [...], 
    --       "sources": [{"url": "...", "title": "...", "type": "museum"}],
    --       "fetched_at": "..."}
  - has_conflicts (bool, 视觉/OCR/检索结果存在冲突)
  - sync_status (text)
  - created_at, updated_at

-- 感受表
impressions
  - id (uuid, PK)
  - exhibition_id (uuid, FK)
  - artifact_id (uuid, FK，可空)
  - voice_local_path (text, 本地语音路径)
  - voice_cloud_url (text)
  - raw_text (text, 转录原文)
  - polished_text (text, LLM 润色后的文本)
  - duration_ms (int, 录音时长)
  - recorded_at (timestamptz)
  - sync_status (text)
  - created_at, updated_at
```

### Storage Bucket 设计（Supabase）
- `photos/{exhibition_id}/{artifact_id}/{filename}` —— 文物原图
- `voices/{exhibition_id}/{filename}` —— 语音文件
- `thumbnails/{exhibition_id}/{artifact_id}.webp` —— 缩略图

### 本地文件管理
- 照片原图存 `FileSystem.documentDirectory + 'photos/'`
- 语音存 `FileSystem.documentDirectory + 'voices/'`
- 缩略图存 `FileSystem.cacheDirectory + 'thumbnails/'`
- 同步成功后允许用户手动清理本地原图（保留缩略图）

## 六、模块拆解与开发顺序

> **严格按顺序开发，每个模块跑通再做下一个。不要并行。**

### 第 1 步｜项目骨架（约 1-1.5 周）

**目标**：跑通 Expo 开发环境、Supabase 连接、基础导航。

**任务**：
1. `npx create-expo-app` 初始化项目，配置 TypeScript + expo-router
2. 安装并配置 NativeWind v4
3. 配置 Supabase 客户端（区分 anon key 和 service key 的使用场景）
4. 实现邮箱登录页 + 受保护的 tab 路由
5. 搭建 4 个 tab 的空壳页面 + 中间凸起按钮
6. 实现"创建展览 / 列出展览 / 删除展览"最小闭环
7. 设置 EAS Build 和 EAS Update 配置文件

**验收标准**：
- 手机用 Expo Go 扫码能跑起来
- 能登录、能创建展览、能在首页看到展览卡片列表
- 关掉 WiFi 重启 App，已创建的展览仍能看到（本地缓存生效）

### 第 2 步｜本地存储与同步引擎（约 1-1.5 周）

**目标**：建立本地优先 + 云端同步的数据基础设施。

**任务**：
1. 用 expo-sqlite 建立本地表结构（与云端一致）
2. 封装数据访问层（DAO），所有读写都先走本地
3. 实现后台同步队列：
   - 应用启动时检查 `sync_status = pending` 的记录
   - 网络可用时按顺序上传到 Supabase
   - 失败重试（指数退避，最多 3 次）
4. 实现拉取同步：登录时拉取云端数据合并到本地
5. 在设置页加"同步状态"指示器（待同步条数）

**验收标准**：
- 飞行模式下创建展览、添加文物、录语音都能正常保存
- 联网后所有内容自动同步到 Supabase
- 在另一台设备登录同账号能看到同样的数据

### 第 3 步｜现场拍摄模块（约 2 周，核心场景）

**目标**：在展厅看完一件文物后 30 秒内完成拍照 + 标注。

**任务**：
1. **自定义相机界面**（expo-camera）：
   - 大快门按钮（拇指可达）
   - 顶部显示当前所属展览（自动选最近创建的，可点击切换）
   - 连拍模式：一次拍多张（文物 + 展签 + 细节）自动归为一组
   - 拍照后短暂震动反馈（expo-haptics）
2. **快速录音**：
   - 拍照后界面下方出现"按住录音"按钮
   - 松开自动保存并绑定到刚拍的文物
   - 录音时显示波形（react-native-audio-waveform 或自绘）
3. **GPS 自动归类**（可选）：
   - 拍照时记录 GPS
   - 如果该展览之前没有 GPS 记录，写入；如果已有且距离 < 200m 则自动归类
4. **新建展览快捷流程**：
   - 相机界面右上角"+"按钮，弹出 modal 输入展览名（其他字段可后补）
5. **拍摄完成回到主流程**：
   - 不强制立刻整理，照片落本地后用户继续看展或离开

**验收标准**：
- 从打开 App 到完成第一张拍照，3 秒内完成
- 单件文物的"拍 3 张照 + 录 10 秒语音"流程，30 秒内完成
- 全程单手操作可达

### 第 4 步｜批量导入（约 0.5-1 周）

**目标**：兼容历史照片和系统相机拍的照片。

**任务**：
1. 在展览详情页加"从相册导入"按钮
2. 用 expo-image-picker 多选照片
3. 读取 EXIF 拿拍摄时间和 GPS（exif 库或 expo-image-picker 自带）
4. 客户端用 expo-image-manipulator 生成缩略图
5. 创建 artifacts 记录写入本地 SQLite，标记为 pending 同步
6. 系统级"分享到 App"入口：相册分享菜单选 App，自动跳到选展览页
   （配置 expo-share-intent，需 EAS Build 才能完整测试）

**验收标准**：
- 一次性选 50 张照片导入，10 秒内完成本地处理
- 从系统相册"分享"过来能正确跳到导入页

### 第 5 步｜OCR + LLM 信息抽取（约 2-3 周，核心价值）

**目标**：从展签照片自动提取结构化信息。

**任务**：
1. 在文物详情页加"识别展签"按钮，单张处理
2. 在展览详情页加"批量识别"按钮，多选照片一起处理
3. 创建 Supabase Edge Function `extract-artifact-info`：
   - 接收 photo_url 数组
   - 对每张图调阿里云 OCR API
   - OCR 文字喂给 LLM，结构化抽取以下字段：
     - 文物名称、朝代、品类、出土地 / 来源、具体年代、完整展签描述
   - 写回 Supabase artifacts 表
4. 移动端通过 Supabase Realtime 订阅 artifacts 表更新，实时刷新 UI
5. 处理完允许在卡片上手动修改任何字段
6. 失败的照片标记为"待手动补全"，不阻塞流程
7. 移动端做"流量保护"：批量识别前提示"将上传 X 张照片到云端，预计消耗 X MB"

**Prompt 模板**（写在 Edge Function 内）：
```
你是博物馆文物信息抽取专家。下面是一段展签 OCR 文字，请抽取结构化信息：

OCR 原文：
{raw_ocr_text}

请输出严格的 JSON 格式，字段如下：
{
  "name": "文物名称",
  "dynasty": "朝代（如：唐 / 宋 / 战国）",
  "category": "品类（如：青铜器 / 陶瓷 / 书画 / 玉器）",
  "origin": "出土地或来源（无则填 null）",
  "era": "具体年代（如：公元前 5 世纪 / 1368-1644）",
  "description": "完整展签描述文字"
}

如果某字段无法判断，填 null。不要编造。只输出 JSON，不要任何其他文字。
```

**验收标准**：50 张展签照片 5 分钟内识别完成，准确率 > 80%。

### 第 6 步｜视觉 AI 识别 + 联网检索增强（约 2-3 周，产品差异化核心）

**目标**：让 App 不仅能"读展签"，还能"看文物"并"查资料"，得出比展签更丰富的文物档案。

**任务**：

1. **视觉识别 Edge Function** `analyze-artifact-vision`：
   - 接收文物本体照片 URL（非展签）
   - 调用视觉 LLM（推荐通义千问 VL-Max，中文文物识别效果最好）
   - 用结构化 prompt 让模型输出：形制 / 主要纹饰 / 推测朝代 / 工艺特征 / 文物名称候选 + 置信度
   - 写入 `artifacts.vision_analysis` 字段

2. **联网检索 Edge Function** `fetch-knowledge`：
   - 输入：文物名（来自 OCR 或视觉识别）+ 朝代 + 品类
   - 调用顺序（找到合适结果即停止）：
     1. 维基百科 API（`https://zh.wikipedia.org/api/rest_v1/page/summary/{title}`）
     2. Tavily Search API，限定 site:dpm.org.cn / chnmuseum.cn 等大馆域名
     3. Tavily 通用搜索 + 百度百科域名
   - 检索到的内容用 LLM 提炼成"扩展知识卡片"（200 字内 + 3-5 条关键事实）
   - 每条信息记录来源 URL
   - 写入 `artifacts.knowledge_card` 字段

3. **三方信息融合逻辑**：
   - 创建 Edge Function `merge-artifact-info`
   - 输入：OCR 抽取结果 / 视觉识别结果 / 联网检索结果
   - 规则：
     - 展签字段（OCR）置信度最高，作为主数据
     - 缺失字段用联网检索补全
     - 视觉识别结果与展签冲突时，标记 `has_conflicts = true`
     - 每个最终字段写入 `field_sources` 标注来源
   - 用户可在文物详情页看到每个字段旁的小标签（"展签" / "视觉 AI" / "故宫官网"等）

4. **UI 展示**：
   - 文物详情页分三个 tab：基础信息（展签）/ 视觉分析 / 扩展知识
   - 信息冲突时顶部红条提示"该文物的 X 字段存在冲突，点击查看"
   - 知识卡片底部显示来源链接，可点击跳转浏览器
   - 用户可点"采纳此版本"覆盖主字段，或手动编辑

5. **成本与流量控制**：
   - 视觉识别**默认不自动跑**，需用户在文物卡片点"深度识别"按钮
   - 提供"批量深度识别"入口，提示预计调用次数和费用估算
   - 设置中可配置"WiFi 下自动跑视觉识别"开关

6. **缓存策略**：
   - 同名文物的联网检索结果在云端缓存 30 天，避免重复调用
   - 视觉识别结果永久缓存（除非用户主动重跑）

**视觉识别 Prompt 模板**：
```
你是一位资深博物馆文物鉴赏专家。请仔细观察这张文物照片，输出结构化分析。

只输出 JSON，不要任何其他文字。如果某字段不确定，填 null。绝不编造。

{

  "estimated_dynasty": "推测朝代（如：商代晚期 / 唐 / 明）",
  "dynasty_confidence": 0.0-1.0,
  "craft_features": ["工艺特征，如 失蜡法 / 釉下彩 / 工笔重彩"],

**联网检索 Prompt 模板**（用于把搜索结果总结成知识卡片）：
```
下面是关于文物「{artifact_name}」的网络检索结果。请提炼成简洁的知识卡片。

要求：
1. 总结不超过 200 字，介绍文物的基本背景、艺术价值、历史意义
2. 列出 3-5 条关键事实（年代 / 出土时间 / 现藏地 / 尺寸 / 重要特征等）
3. 严格基于提供的资料，不要补充任何资料外的信息
4. 如果资料明显有误或互相矛盾，标注出来

输出 JSON：
{
  "summary": "200 字内总结",
  "key_facts": ["事实 1", "事实 2", ...],
  "conflicts_or_uncertainties": "若有冲突或不确定，描述；否则填 null"
}

{search_results}

**验收标准**：
- 选 10 件常见文物（如台北故宫翠玉白菜、国博后母戊鼎），视觉识别能准确判断形制和大致朝代
- 联网检索能从博物馆官网或维基百科获取扩展信息
- 三方信息融合后，最终展示无明显错误，每个字段可溯源

### 第 7 步｜语音转文字 + 自动绑定（约 1 周）

**任务**：
1. Edge Function `transcribe-voice`：调用通义听悟 / Whisper 转录
2. 自动绑定：根据语音录制时间戳找到时间最接近的 artifact，绑定 impression.artifact_id
3. 文物详情页显示转录文本，支持播放原音频
4. 长按 impression 卡片可"改绑到其他文物"

**验收标准**：转录准确率高，自动绑定准确率 > 70%。

### 第 8 步｜整理与归档（约 1.5-2 周）

**目标**：把碎片化内容组织成完整观展档案。

**任务**：
1. 展览详情页两种视图切换：
   - 时间顺序视图（瀑布流缩略图）
   - 策展单元视图（按 curatorial_units 分组）
2. "策展叙事生成"按钮：把展览名 + 文物列表（含联网检索的扩展知识）喂给 LLM，生成 3-6 个策展单元
3. 用户可手动调整单元划分（拖拽 / 长按菜单）
4. 自动把 artifacts 分配到最匹配的单元
5. "感受润色"功能：把同一展览所有 raw_text 喂给 LLM，生成连贯整体感受
6. 单件文物 / 整体感受都支持"原始 / 润色"切换

**验收标准**：识别完的展览 3 分钟内生成完整策展单元 + 润色感受。

### 第 9 步｜文博数据库与统计（约 1 周）

**目标**：跨展览搜索与统计，但保持移动端轻量。

**任务**：
1. Library tab：
   - 顶部搜索框（同时搜文物名 / 朝代 / 品类 / 感受 / 知识卡片内容）
   - 横向滚动筛选 chip（朝代 / 品类 / 博物馆 / 年份）
   - 结果按瀑布流展示
2. Profile tab：
   - 看过多少场展览、多少博物馆、多少件文物
   - 朝代分布饼图（victory-native 或 react-native-svg-charts）
   - 月度观展柱状图
3. 搜索用 SQLite 的 FTS5（本地全文搜索，秒级）
4. 复杂统计如"年度报告"留 v2

**验收标准**：搜"唐代 青铜器"秒级返回所有相关文物。

### 第 10 步｜导出与分享（约 1 周）

**任务**：
1. **Markdown 导出**：单展览导出为结构化 markdown
2. **JSON 备份**：全库导出为 JSON 文件
3. **图文长图**：用 react-native-view-shot 截屏生成"小红书风格"分享图
4. **分享到系统**：用 expo-sharing 调起系统分享菜单（微信 / 微博 / 保存到相册）
5. **PDF 导出**：用 expo-print 生成 PDF（基于 HTML 模板）

**验收标准**：一场展览能 5 秒内生成各种格式，调起系统分享菜单。

## 七、UI / UX 准则

- **视觉风格**：克制、博物馆质感，参考小红书的图片墙 + Notion 的结构化布局
- **配色**：白底 + 深灰文字 + 一个主题色（推荐黛蓝 #2C3E50 或墨绿 #1F4D3C）
- **字体**：使用系统中文字体，标题加粗
- **触控目标**：最小 44pt，关键按钮 56pt+
- **手势**：
  - 列表项左滑显示删除 / 编辑
  - 长按卡片进入多选模式
  - 下拉刷新同步
- **反馈**：所有操作配合 expo-haptics 触感反馈
- **加载状态**：用骨架屏（skeleton）而不是 spinner
- **空状态**：每个空列表都要有友好引导文案 + 主操作按钮
- **暗黑模式**：v2 再做

## 八、AI 协作约定（重要）

每次让 AI 写代码时，请遵守以下约定：

1. **每次只做一个模块的一个小任务**，不要一次让 AI 写完整模块
2. **先要求 AI 列出实现计划**，确认没问题再让它写代码
3. **生成代码后，先在手机上验证，再决定是否合并**
4. **报错处理**：把 Expo Dev Tools 完整报错（含堆栈）截图或复制贴回对话
5. **避免一次性大重构**：宁可分 5 次小改，不要让 AI 一次改 10 个文件
6. **数据库迁移**：每次改表结构都让 AI 同时给出本地 SQLite 迁移和云端 Postgres 迁移
7. **环境变量**：所有 secret（Supabase keys / OCR Key / LLM Key）放 `.env` 并用 `expo-constants` 读取，绝不提交 Git
8. **类型先行**：每个数据结构先定义 TypeScript 类型，再写实现
9. **代码风格**：函数式优先、避免 class、避免过度抽象
10. **原生模块谨慎引入**：每加一个非 Expo 官方库，确认它支持 Expo 托管工作流（看 `expo install` 是否能装）

## 十、风险与降级方案

| 风险点 | 降级方案 |
|--------|---------|
| OCR 识别率低 | 手动编辑文物字段，UI 必须支持快速批量编辑 |
| LLM 抽取不准 | 显示原始 OCR 文字 + 抽取结果对照，允许一键回退 |
| **视觉 AI 识别错误（如把战国铜鼎认成商代）** | 标记为低置信度，以展签为准；用户可一键回退 |
| **视觉 AI 调用成本高** | 默认手动触发，不自动跑；批量调用前提示费用估算 |
| **联网检索找不到该文物** | 优雅降级：仅展示 OCR + 视觉识别结果，不报错 |
| **百度百科 / 博物馆官网反爬** | 优先用维基百科官方 API + Tavily 等专业搜索 API |
| **检索结果质量差（搜索引擎返回无关网页）** | 限定 site: 大馆域名 + LLM 二次过滤 |
| **AI 幻觉混入档案** | 强制每个字段标注来源，UI 可视化区分；冲突时不自动覆盖 |
| 语音转录失败 | 允许直接打字输入感受 |
| 策展单元生成不合理 | 提供"重新生成"按钮 + 手动编辑 |
| Supabase 免费额度超限 | 监控用量，必要时清理旧照片或升级到 Pro |
| 手机存储被照片占满 | 设置中加"清理已同步原图"功能 |
| 展厅信号差 | 离线优先架构保证基础功能可用，AI 识别延后到联网时 |
| Expo SDK 升级破坏依赖 | 锁定 SDK 版本，每次升级前看 changelog |
| 真机权限问题（相机 / 录音 / 相册）| 首次使用引导动画，权限拒绝后给清晰的引导文案 |

## 十一、明确不做的功能（避免范围蔓延）

以下功能即使你看到觉得"加一下也不难"，**MVP 阶段坚决不做**：

- ❌ iOS 版（除非你有 Mac 和苹果开发者账号）
- ❌ 平板大屏适配
- ❌ 拍摄时玻璃反光自动优化（物理偏振镜更有效）
- ❌ 多人协作 / 共享档案
- ❌ 社交关注 / 评论 / 点赞
- ❌ 推送通知（v2）
- ❌ 暗色模式（v2）
- ❌ AI 推荐"你可能感兴趣的展览"
- ❌ 商业化相关（订阅 / 付费 / 广告）
- ❌ 国际化多语言
- ❌ 复杂权限系统
- ❌ 端上跑 LLM / OCR 模型（电量和包大小都不允许）
- ❌ 视频录制
- ❌ AR 功能
